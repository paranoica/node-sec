# Bedrock — инструкции (RU)

Всё, что делает шаблон, как работает каждая часть, **как правильно этим пользоваться** и полный
каталог edge cases. Краткую версию см. сперва в [OVERVIEW.ru.md](OVERVIEW.ru.md).

---

## 0. Поток

1. **Use this template** на GitHub → клон → открыть в своём агенте.
2. Сказать, что хочешь построить, или запустить `/genesis`.
3. genesis проводит интервью и пишет спеку + план + правила + первую карту, и **заменяет заглушку
   `README.md`** на README твоего проекта.
4. Дальше каждое изменение проходит через нужный гейт. Ты говоришь — файлы синхронны.

---

## 1. Как правильно пользоваться (do / don't)

**Do**
- **Запускай genesis первым** на новом или незапланированном проекте — до кода. Это фронт-дверь.
- **Отвечай на интервью честно, включая «не знаю».** «Не знаю» — это фича: становится записанным
  открытым вопросом, а не догадкой.
- **Прочитай `docs/` и `PLAN.md`** после genesis — это твоя спека и план; сверь с замыслом до сборки.
- **Строй через цикл:** `backlog.py next` → одна задача → нужный гейт → `backlog.py done <id>`. По одной.
- **Меняешь спеку — переразбирай.** Правишь `docs/` (или добавляешь фичу через `replan`), затем
  `backlog.py re-derive` — дай плану переразбиться. План руками не патчишь.
- **Масштабируй гейт по риску:** тривиальный диф → лёгкое ревью; auth/деньги/миграции/крипто → полный.
- **Гоняй `bash tools/run-evals.sh`** перед правкой собственного тулинга/скиллов шаблона.

**Don't**
- **Не правь `PLAN.md`/`genesis.tasks.json` руками** — состояние только через `backlog.py` (ручная
  правка видимо валит `validate`).
- **Не переписывай универсальную секцию `AGENTS.md`** — правила проекта живут в её секции «Project
  rules» (inline), которую заполняет genesis.
- **Не корми design-creator hook'ом/нарративом** — давай структурные ограничения; hook он формирует сам.
- **Не доверяй устаревшей карте.** Сначала `build.py --check`; устарела → пересобрать.
- **Не давай фиче попасть в код мимо бэклога** в genesis-managed проекте — проводи через `replan`,
  чтобы спека и план оставались источником правды.

## 2. genesis — инициация и (ре)планирование

Фронт-дверь. Владеет «начать/спланировать проект». **Не** ревьюит код (это code-review) и **не**
проектирует UI (это design-creator).

### Режимы
- **greenfield** — пустой репо, новый проект → полный конвейер.
- **adopt** — существующий репо без спеки/правил → genesis реверс-выводит скелет спеки, правила и
  карту **не ревьюя код**.
- **replan** — спека изменилась **или запрошена новая работа** (например из prompt-refiner) →
  переразбор бэклога. Для новой работы replan **сперва амендит спеку** (таргетное интервью по дельте →
  заякоривает), потом переразбирает.

### Интервью
Глубина по сложности (крошечная задача → пара вопросов; многоповерхностный продукт → глубоко).
Спрашивает **по одному открытому вопросу** (дискретные форки можно группировать в пределах темы) и
**никогда не выдумывает решение, которого ты не принял** — неизвестное → `TODO(decision: …)` в
`docs/open-questions.md`. В начале фиксирует **project-type** (web-app / service / CLI / library /
worker) — он решает, эмитить ли design-brief (только визуальный веб) и как настроить шаги CI. Ближе к
концу задаёт **один сгруппированный вопрос — какими агентами пользуется команда** (чтобы эмитить нужные
обёртки; дефолт = Claude + `AGENTS.md`). Скажешь «делай как лучше» — пойдёт на дефолтах, но минимум всё
равно вытянет.

### Что производит (твой источник правды)
- `docs/decisions.md` · `architecture.md` · `glossary.md` · `open-questions.md` — спека.
- `PLAN.md` (человеку) + `genesis.tasks.json` (машине) — план задач.
- `AGENTS.md` — правила для всех агентов; genesis заполняет его секцию **«Project rules»** inline. Плюс
  пер-агентные обёртки (`CLAUDE.md` = `@AGENTS.md`; остальные — под выбранных агентов).
- `README.md` — README твоего проекта (заменяет заглушку Bedrock).
- `.map/project.json` — первая структурная карта.
- `.github/workflows/ci.yml` + `spec-gate.yml` — реальный рабочий CI (см. **CI** ниже).
- `.gitignore` — собран из github/gitignore (твой стек) + блок артефактов скиллов bedrock.
- `project-context/` — сырое интервью + саммари (история, **не** источник правды).

### Как переразбор остаётся честным (механизм якорей)
Каждая атомарная единица спеки (решение, термин глоссария, инвариант архитектуры) несёт стабильный
**якорь**; каждая задача хранит, из каких якорей выведена (`spec_refs`), с контент-**хэшом**. При
изменении спеки `backlog.py re-derive` сверяет хэши и помечает задачи — точное поведение
`needs-review` vs `stale` см. в каталоге edge cases (§10). `done` сохраняется.

### Гейт перед «готово» (spec-analyze)
Два слоя: **детерминированный** чек (каждая задача трассируется в реальный якорь спеки — висячая
ссылка = CRITICAL; orphan-decision без задачи = LOW-флаг; нет противоречий якорей; частичная разметка /
дубли id валят) **плюс** свежеконтекстный **spec-verifier**, читающий спеку с нуля и судящий о более
сложном вопросе *покрытия* — не упущена ли задача для чего-то нужного? — что детерминированный проход
доказать не может. Гейт блокирующий — «готово» не объявляется на пропущенном **или устаревшем** гейте
(рецепт `--check` должен быть fresh). Детерминированная половина + согласованность бэклога **также
форсятся в CI самого проекта** (`.github/workflows/spec-gate.yml`, чистый Python, без модели) — так что
«блокирующий» держится механически, а не только моделью; свежеконтекстный verifier — половина авторинга.

### CI — genesis пишет workflow проекта
genesis эмитит **реальный рабочий** `.github/workflows/ci.yml` (lint / test / build), выбранный по
**stack × project-type** — Node / Python / Go / Rust, либо `generic`-скелет с TODO для нераспознанного
стека (**никогда не фейк-грин CI**). **В CI не исполняется AI** — это обычные джобы GitHub на его
раннерах. Также добавляет бэклог-таску «Wire up CI», чтобы «сделать CI зелёным» был отслеживаемой
единицей, а не непроверенным артефактом. Единственное обращение к сети — опциональное разовое
обновление пинов версий при генерации; **офлайн → берёт дефолтные пины и оставляет комментарий
`# verify pins`**. Спутник `spec-gate.yml` (детерминированная половина гейта, выше) гоняется на каждом
PR. Это CI *проекта* — собственный самотест Bedrock'а это `tools/run-evals.sh` (§8), другой файл.

### .gitignore — собран, не написан руками
genesis собирает `.gitignore` проекта из **официального [github/gitignore](https://github.com/github/gitignore)**
(CC0): универсальный базис `common` (OS / редакторы / `.env*`-секреты / логи / temp) + фрагмент(ы) под
твой **стек** (Node / Python / Go / Rust; неизвестный стек → `common` + TODO, никогда не «фейк-полный»
ignore) + собственный блок артефактов скиллов bedrock (`.map/`, `.genesis/`, …), склеено в этом
порядке. Фрагменты вендоренные (работает офлайн), с опциональным разовым refresh из источника.
**Лок-файлы никогда не игнорятся** (коммитятся для воспроизводимых установок); `.vscode/` не режется
наглухо; существующий `.gitignore` расширяется, не перезаписывается.

### Честность режима adopt
В adopt genesis видит **что** делает код, но не **почему**. Наблюдаемые факты → `architecture.md`
**со ссылками file:line**; реверс-выведенное обоснование → `open-questions.md` как
`inferred/unconfirmed` — **никогда** не утверждается как решённое, пока ты не подтвердишь.

## 3. prompt-refiner — ловец остатка

Ловит мутные запросы **только когда их не взял ни один другой скилл** и превращает в точный промпт.

- **Когда срабатывает vs молчит (residue-тест):** тест — **«может ли профильный движок начать как
  есть?»**, не тема. Резолвится в один движок (пусть размыто *внутри* — движок доспросит сам) →
  маршрутизирует туда, **молчит**. Не резолвится (двоится между движками или нет цели) → **остаток →
  берёт**, точит, маршрутизирует.
- **Что делает, когда берёт:** находит цель инструментами (а не спрашивает), пишет точный промпт (одна
  задача, файлы, acceptance, verify-handle, ссылка на паттерн), задаёт **≤1** уточнение только при
  материальном расхождении, маршрутизирует. Точит молча; **отменяем одним словом**.
- Managed-проект и дефолт-уступку при сомнении см. в §10.

## 4. project-map — общая карта кода

`tools/project-map/build.py` обходит репо и пишет `.map/project.json`: файлы → символы (определяет/
вызывает + обратные рёбра) и best-effort **доменные срезы** (routes, data-model, FSM, queues).

- **Свежесть (суть):** карта несёт штамп. Перед использованием: `build.py <root> --check` →
  `fresh` / `stale` / `absent` (exit 0/1/2). `stale` → инкрементальный ребилд (`build.py <root>`).
  **Устаревшая карта никогда не выдаётся за факт.**
- **Лиды, не факты:** каждое ребро и срез несут `file:line` + уверенность; открой и подтверди. Edge
  cases low-confidence / absent / незнакомый стек — в §10.

## 5. design-creator & code-review (вендоренные движки)

- **design-creator** проектирует/строит веб-UI. Для **визуально-веб project-type** genesis отдаёт ему
  **структурный бриф** (`.genesis/design-brief.json`: домен, аудитория, поверхности, scope, тон,
  бренд-ассеты) — **без hook и нарратива** (схлопнуло бы разнообразие); CLI / service / library /
  worker брифа не получает. Бриф потребляет агент, вызывающий design-creator (в его анкету), а не
  автономный ридер файла.
- **code-review** аудитит существующий код. Мандат — цикл **implement → review → apply fixes →
  re-review**, масштаб по риску.

## 6. Канон, цикл и обёртки

`AGENTS.md` — **канонический** документ правил (его читают нативно большинство агентов). `CLAUDE.md` —
тонкая обёртка `@AGENTS.md` для Claude Code. Оба кодируют **мандаты гейтов** (дизайн→design-creator;
аудит→цикл code-review; всё прочее→цикл ниже), **status seam** и **map-read protocol**. Цикл остаточной
работы:

1. `backlog.py next` → следующая готовая задача.
2. Делаешь одну (в её `files`, удовлетворяя `acceptance`).
3. Запускаешь нужный гейт по риску.
4. `backlog.py done <id>`.
5. Перечитываешь инварианты на границе, цикл повторяется.

Твои **правила проекта** живут inline в секции «Project rules» файла `AGENTS.md` (genesis пишет их там,
чтобы их видел каждый агент — не за Claude-only `@import`).

## 7. Работа с разными агентами (агент-агностичность)

`AGENTS.md` — кросс-агентный стандарт, его читают **нативно** Cursor, Roo Code, Windsurf и Codex — без
склейки. Для остальных genesis эмитит тонкую обёртку под выбранных в интервью агентов:

| Агент | Что нужно |
|-------|-----------|
| Claude Code | `CLAUDE.md` (`@AGENTS.md`) — уже есть |
| Cursor / Roo / Windsurf / Codex | ничего — читают `AGENTS.md` нативно |
| Aider | одна строка в `.aider.conf.yml`: `read: [AGENTS.md]` |
| Continue | генерируемый `.continue/rules/00-bedrock.md` (из `AGENTS.md`) |
| **Antigravity** | `.agents/rules/bedrock.md` = `@/AGENTS.md` (Always On) — путь workspace-правил; глобальный `~/.gemini/GEMINI.md` — юзерский, не трогаем |

Single source: каждая обёртка указывает на / зеркалит `AGENTS.md`; правила в обёртке не повторяются.
(Про скиллы: скиллы — открытый стандарт (agentskills.io), которому Codex тоже конформен, но грузит их
из `.agents/skills/`, а не `.claude/skills/`. Значит они **частично портируемы** на Codex: запусти
`python3 tools/port-skills.py <skill>` — он зеркалит папку в `.agents/skills/` и снимает
Claude-специфичный frontmatter (`allowed-tools`, `${CLAUDE_SKILL_DIR}`, `$ARGUMENTS`). Портируй
consume-side скиллы (prompt-refiner, code-review, design-creator) — **не** genesis, который
исполняется в Claude Code. Проверено по докам OpenAI Codex.)

## 8. Инструменты и самопроверки (для работы над самим шаблоном)

- `tools/project-map/build.py` — карта.
- `.claude/skills/genesis/scripts/backlog.py` — бэклог-тулза (`stamp`/`next`/`done`/`status`/
  `re-derive`/`validate`/`render`).
- `tools/contract.json` — **единый источник** межскилловых путей, команд и фактов о якорях.
- `tools/drift-check.py` — падает, если машинный регион `AGENTS.md`, контракт карты и скрипты genesis
  **расходятся** (а не только если файл отсутствует). Четыре связки: map-path, gate-commands,
  anchor-facts, design-brief.
- `tools/port-skills.py` — зеркалит скилл в `.agents/skills/` для Codex (`<skill>` или `--all`),
  снимая Claude-специфичный frontmatter. Копия — генерируемое зеркало; перегоняй после правки источника.
- `tools/run-evals.sh` — одна команда, гоняющая **все** регрессы и падающая, если упал **любой**. CI
  (`.github/workflows/evals.yml`) гоняет то же.

```bash
bash tools/run-evals.sh
```

## 9. Что коммитится vs пересобирается (не сломай)

- **Коммитится** (источник правды + история): `docs/`, `PLAN.md`, `genesis.tasks.json`, `AGENTS.md`,
  `CLAUDE.md` (+ любые пер-агентные обёртки), `README.md`, `.github/workflows/` (сгенерированный CI),
  `project-context/`, плюс выученное состояние вендоренных скиллов (`.design/tokens.json`,
  `.review/suppressions.json`, …).
- **Git-ignored** (пересобираемое / локальное): `.map/`, `.genesis/`, `.refiner/`, `.design/mockups/`,
  `.review/index.json`, `.review/outcomes.jsonl`.

## 10. Edge cases и подвохи — полный каталог

**Интервью**
- *Не можешь ответить* → пишется `TODO(decision: …)` в `open-questions.md`; genesis не угадывает. Решишь
  позже (мигрирует в `decisions.md` с тем же id).
- *Говоришь «делай как лучше»* → разумные дефолты, но минимум (scope, ключевые сущности, money custody
  если есть деньги) всё равно вытягивается — на нуле не работает.

**Изменение спеки → re-derive** (`backlog.py re-derive`)
- *Чисто форматная правка* (заболдить, перенести строку) → **нет дрейфа**. Хэш игнорит форматирование.
- *Правка формулировки решения* → задачи на него → **`needs-review`** (мягко: подтверди валидность).
- *Переименование/удаление якоря, от которого зависит задача* → задача → **`stale`** (переразобрать).
- *Затронута `done`-задача* → **`needs-review`**, никогда не «раз-done» молча.
- *Меняешь термин глоссария* → каждая задача, зависящая от него *через* решение, помечена
  (**транзитивно**), даже без прямой связи задача→термин.
- *Задача зависит от нерешённого `TODO(decision: …)`* → гейт помечает её **не готова к исполнению**,
  пока решение не принято — неизвестное блокирует зависимую работу.

**План / status seam**
- *Правишь `PLAN.md`/`genesis.tasks.json` руками* → `backlog.py validate` ре-рендерит из источника и
  диффит; рассинхрон — **видимый ненулевой провал**, не тихий дрейф. Фикс: `backlog.py render` (и меняй
  состояние через `backlog.py`, не руками).
- *Зависимости задачи не done* → `backlog.py done <id>` откажет (deps сперва).

**Карта**
- *`--check` = `stale`* → ребилд (`build.py <root>`); перепарсятся только изменённые файлы.
- *`absent` и нельзя пересобрать* (нет Python, read-only FS) → скажи и ограничь анализ прочитанными
  файлами. Устаревшую за факт не выдавай.
- *Срез `low` confidence или `absent`* → это **лид, не факт**; открой `file:line` и подтверди. `absent`
  = «лид не найден», не «ничего нет».
- *Незнакомый стек/фреймворк* → ложный срез не выдумывается (срез просто absent).

**Скиллы и маршрутизация**
- *Запрос может быть дизайном или кодом* → маршрутизация детерминирована по `AGENTS.md`:
  дизайн/редизайн → design-creator; аудит существующего → code-review; prompt-refiner срабатывает
  **только** на остатке, который не взял никто, и **уступает при сомнении** (ложное молчание дёшево —
  движок доспросит сам).
- *Фича-реквест в genesis-managed проекте* → идёт в **genesis replan** (амендит спеку → бэклог), а не
  прямо в код. В non-managed репо prompt-refiner точит напрямую.
- *Перезапуск genesis на существующем проекте* → он **никогда не перезаписывает** универсальную секцию
  `AGENTS.md`, обёртку или твою работу — читает-и-расширяет и выносит конфликты.

**Мультиагентность**
- *Antigravity* → **supported через `.agents/rules/`**: genesis эмитит `.agents/rules/bedrock.md` =
  `@/AGENTS.md` (Always On). Корневой `AGENTS.md` он как rules НЕ читает (исправленный ранний claim);
  `~/.gemini/GEMINI.md` — глобальный юзерский файл правил, genesis его не трогает.
- *Скиллы Codex* → **частично портируемы (проверено)**: Codex грузит скиллы из `.agents/skills/`,
  поэтому запусти `tools/port-skills.py <skill>` — зеркалит + снимает Claude-специфичный frontmatter.
  Портируй consume-side скиллы, не genesis.

**Провалы гейта (spec-analyze)**
- *Частичная разметка* (часть единиц заякорена, часть нет, в must-anchor файле) → **CRITICAL**.
- *Висячая ссылка* (`spec_refs`/`refs:` id без якоря) → **CRITICAL**.
- *Дубль anchor id* → **CRITICAL**.

**CI (сгенерированный workflow)**
- *Офлайн при генерации* → genesis берёт дефолтные пины версий и оставляет комментарий `# verify pins`;
  на сеть он не блокируется.
- *Нераспознанный стек* → эмитит скелет `generic.yml` с TODO — **никогда не фейк-грин CI**, проходящий
  без реального прогона.
- *Сгенерированный workflow ≠ проверенный* → реальные шаги `ci.yml` гоняются только когда есть код,
  поэтому **первый засиженный проект обязан показать зелёный прогон Actions** («verify on first real
  seed»). `spec-gate.yml` — чистый Python, проверен локально.

**.gitignore (собранный)**
- *Неизвестный стек* → `common` + блок bedrock + маркер `# TODO(genesis)` — никогда не «фейк-полный».
- *Лок-файл* (`package-lock.json`, `uv.lock`, `Cargo.lock`, …) → **никогда не игнорится** — коммитится
  для воспроизводимых установок.
- *У тебя уже есть `.gitignore`* → genesis его расширяет (дописывает недостающие секции), не перезаписывает.

## 11. Подвохи для мейнтейнеров

- **`normalize()` — это миграция, не фикс.** Хэш якоря считает одна функция
  (`scripts/anchors.py: normalize()`), общая для генератора и гейта. Её изменение пере-хэширует
  **каждый** якорь в **каждом** проекте, созданном из шаблона — версионируй осознанно; не патчь тихо.
- **У `.gitignore` нет trailing-комментариев.** `# …` на строке паттерна становится частью паттерна и
  тихо ломает игнор. Комментарии — на своих строках.
- **Калибровка холодна долго** (копится по одному проекту); flip-cause-петля **по умолчанию не
  заряжает** планку интервью — «пользователь передумал» никогда не считается провалом интервью.
