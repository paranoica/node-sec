# 3D

The 3D block. Stack is **always React Three Fiber + drei** (projects are always React/Next). 3D is **always confirmed via the survey** — never a silent default.

## Anatomy of a scene (the vocabulary)

Any R3F scene is six parts:
1. **`<Canvas>`** — the root; all 3D lives inside it; it sits in the React page like any element — the bridge between 2D and 3D.
2. **Camera** — the point of view; most often what is bound to scroll.
3. **Mesh — the object** = **geometry** (the shape) + **material** (the surface). Two distinct layers: geometry is *what* shape, material is *how* it looks in light.
4. **Light** — without it the mesh is invisible or flat. Half of the "wow", and pure code.
5. **Environment** — an HDRI map; sets reflections, background, soft overall light. One line in drei.
6. **Render loop (`useFrame`)** — runs each frame; all motion lives here (rotation, scroll-binding, animation).

Minimal scene = Canvas + camera + mesh + light. Environment and render loop add realism and motion.

## Materials (six) — ~80% of an object's visual character

- **Glass — `MeshTransmissionMaterial`** (drei). Max "wow"; the Igloo look. Expensive — 1–2 per scene.
- **Metal — `meshStandardMaterial` `metalness=1`.** Reflects the environment — **needs an HDRI environment or it is dead** (a mirror with nothing to reflect).
- **Matte — `meshStandardMaterial` high roughness.** The workhorse; cheap; when the form matters more than spectacle.
- **Emissive — `emissive` + `emissiveIntensity`.** Glows from within; with Bloom postprocessing it becomes a neon source. Rhymes with the site's accent color.
- **Iridescent — `meshPhysicalMaterial` `iridescence`.** Shifts color by angle (soap-film / pearl). Very showy; the heaviest standard material; works only in motion.
- **Wireframe — `meshBasicMaterial wireframe`.** Techno look; almost free on GPU; an accent or a background layer.

Two rules: effective materials (glass, iridescent, emissive) are an **accent — 1–2 per scene** (intensity by hierarchy). Metal and glass **require an Environment (HDRI)** — material and environment go as a pair.

## Lighting

- Three-point setup (key shapes the form, fill softens shadows, rim separates from the background) — for matte objects.
- Environment / HDRI as the light source — for metal and glass.
- Often a combination. Light is half the scene quality and cheap on GPU.

## Interaction (the "react to the user" part)

- **Hover on the object** — `onPointerOver`/`onPointerOut`; the object reacts like a button (slight scale, emissive highlight, pointer cursor). Anything clickable must respond.
- **Scroll drives the scene** — `ScrollControls` + `useScroll` (drei); scroll position drives rotation / camera. The Igloo / Active Theory technique. A scroll-driven 3D scene behaves like a pin section with three-dimensional content. **Scroll-controlled 3D only via a proven library** (drei `ScrollControls` / GSAP) — never hand-rolled.
- **Cursor reaction** — the object eases toward the pointer via `lerp` in `useFrame` (lerp is mandatory — without it the object jitters).

## Postprocessing

`@react-three/postprocessing` — effects over the whole picture. **Bloom** is the main one — turns an emissive material into a neon source; half the award look rides on it. DepthOfField, ChromaticAberration, Vignette — dosed. Postprocessing is GPU-heavy — trim or disable on mobile.

## Three levels of 3D (a survey question)

- **Light** — one 3D object in the hero, the rest 2D. Cheap, safe, a visible "wow" for little cost — the default for most.
- **Through-line** — a 3D object bound to scroll, passing through sections, the camera moving — 2D and 3D stitched together. Costlier; needs a scenario.
- **Experience** — the whole site is a 3D scene (Active Theory level). Maximum wow and maximum cost: performance, assets, art direction, time.

The engine states the cost of each level honestly; the user chooses.

## Asset pipeline

The bottleneck of 3D is not code — it is assets and art direction.
1. User has a reference / ready model → use it.
2. None → look for a ready model (Sketchfab / Poly Haven stocks; Meshy / Luma / Rodin generators).
3. None and none found → **fork by object type**: an abstract / parametric form (sphere, torus, crystal, wave, distorted primitive) → build it in code (R3F). A concrete recognizable object (a person, a car, a realistic animal, a shoe, a building) → say honestly "this needs to be modeled or generated, code will not do it" — never lather up a surrogate from primitives.

## Cross-cutting

Everything heavy (glass, postprocessing, heavy materials) must have a mobile fallback and a performance check. `prefers-reduced-motion` disables auto-rotation and abrupt movement. Cursor-dependent interaction is disabled on touch.

## Degradation ladder

Heavy techniques define their own fallbacks as part of the technique, not as an afterthought. Specify, top to bottom: full experience → reduced-motion alternate (designed static, `motion.md`) → no-WebGL / low-GPU path → touch/no-hover path → no-JS baseline. The page is correct and on-brand at every rung. A technique whose fallback wasn't designed isn't finished.

## Implementation stack (concrete — so it actually gets built)

The aesthetic above is useless without the tooling to ship it. Default stack, installed in
the **project** (not the skill):

```bash
npm i three @react-three/fiber @react-three/drei @react-three/postprocessing
npm i gsap lenis            # scroll-driving the scene (see storytelling.md)
# optional: leva (debug controls), maath (easing/math), three-stdlib
```

- **three + @react-three/fiber + @react-three/drei** — the core. drei gives you `Environment`,
  `MeshTransmissionMaterial`, `ScrollControls`/`useScroll`, `useGLTF`, `Float`, `Center`, etc.
  for one line each. **Reach for drei before hand-rolling.**
- **@react-three/postprocessing** — Bloom/DOF/etc. (half the award look rides on Bloom).
- **TSL (Three.js Shading Language)** when you want node materials that compile to **both
  WebGL and WebGPU** — the current path for forward-looking shader work. Plain GLSL
  (`shaderMaterial` from drei) is still fine and simpler.
- **OGL** — a lighter alternative to three when the scene is a few shader planes, not a world
  (faster load, less API). Use it when three is overkill.
- **Spline** — no-code 3D you embed; fast results, less control. Fine for a single hero object
  when bespoke shader work isn't warranted.

**The procedural-first default (this is the implementation fix).** When there's no supplied
model, the *ambitious default is not a placeholder* — it's **procedural geometry + a custom
shader**, which is pure code Claude writes well: distorted/animated primitives, particle
fields, instanced meshes, raymarched SDFs, flow/noise fields, a displaced plane, a point
cloud. These read as "real WebGL craft", not a stand-in. Only fall back to "this needs a real
model" for a **concrete recognizable object** (a person, a specific car, a realistic animal) —
per the asset-pipeline rule below. Reaching for procedural+shader instead of bailing is the
difference between an Active-Theory-tier hero and an empty box.

## Asset sources (CC0 / permissive — vetted, with license discipline)

When a real model, HDRI, or texture is needed, pull from sources that are actually licensed
for use, and record the license + attribution in `.design/journal.md`:

- **HDRI / environment maps:** Poly Haven (CC0), ambientCG (CC0). An HDRI is what makes metal
  and glass materials live — always grab one for those.
- **Textures / PBR materials:** Poly Haven (CC0), ambientCG (CC0).
- **Models:** Khronos glTF-Sample-Assets (varied, check each), Quaternius (CC0 low-poly sets),
  Poly Haven (CC0 models), Sketchfab **filtered to CC/CC0** (verify per-model). Format: prefer
  **glTF/GLB**, compressed with **Draco** or **Meshopt** (drei `useGLTF` supports both).
- **Generators (when nothing fits):** Meshy / Luma / Rodin — output still needs cleanup +
  retopo for web; treat as a starting point, and be honest about the quality ceiling.

Never ship an asset whose license you didn't check, and never claim an asset is CC0 without
verifying — fabricating a license is a `governance.md` violation.

## Performance discipline (from real award sites, not theory)

Award-winning WebGL ships smooth because of explicit discipline, not luck:

- **Lazy-load the 3D.** Don't block first paint on the canvas; mount it after the shell, or on
  scroll-into-view. Code-split the three bundle.
- **Pause the render loop when the canvas isn't visible** (off-screen, tab hidden, or a
  project/detail page is open) — `frameloop="demand"` or gate `useFrame`. Idle GPU = wasted
  battery and heat.
- **Throttle expensive work** — raycasting at ~30fps, not every frame; cap `dpr` (`dpr={[1,
  2]}`) so retina doesn't 4× the fragment cost.
- **Instancing** for many repeated meshes (`InstancedMesh`); never N separate meshes in a loop.
- **Compress** geometry (Draco/Meshopt) and textures (KTX2/basis); resize textures to what's
  actually shown.
- **Mobile**: trim postprocessing, drop to a lighter material, lower `dpr`, or fall to the
  degradation ladder's reduced path. Test at a real 320px + mid-tier GPU, not just desktop.

These hang off the existing `perf-budget.md` budget; a 3D technique that blows the budget is
either trimmed to fit or surfaced as an explicit trade-off, never shipped silently heavy.
